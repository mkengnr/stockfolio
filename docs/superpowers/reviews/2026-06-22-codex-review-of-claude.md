# Codex 교차검토: Claude 시장별 종가·스냅샷 개선안

검토 대상: `2026-06-22-market-snapshot-claude-proposal.md`

## 총평

Claude 안은 `price_date` 무시, 장중 백필, 기존 날짜 미갱신, APScheduler 미스파이어, Redis/DB 출처 분리, 전역 최소 비교일이라는 핵심 원인을 정확히 찾았다. 특히 모든 스냅샷 쓰기를 하나의 확정 진입점으로 모으자는 제안과 DB를 진실의 원천으로 두는 방향은 타당하다.

다만 시장별 실행 시각, 전체 과거 데이터 복구, 실제 동시성, DB/Redis 쓰기 순서가 충분히 구체화되지 않았다. 이 상태로 구현하면 일부 완료 조건을 충족하지 못할 수 있다.

## 채택할 강점

1. 저장 날짜는 항상 제공자의 `PriceResult.price_date`를 사용한다.
2. 장중 당일 바는 확정 스냅샷으로 기록하지 않는다.
3. 백필과 종가 확정을 분리하고 모든 확정 쓰기를 공통 함수로 통합한다.
4. cron에 `misfire_grace_time`, `coalesce`, `max_instances`를 명시한다.
5. 시작 시 마감이 끝난 시장의 누락 종가를 복구한다.
6. 종가 확정 결과를 PostgreSQL과 Redis에 동일하게 반영한다.
7. `updated_at`과 구조화 로그로 관측성을 추가한다.
8. 데이터 복구는 Alembic이 아닌 dry-run 기본 CLI로 수행한다.
9. 대시보드 복구 기준을 종목별 `quote.price_date`로 바꾼다.

## 보완이 필요한 사항

### [High] C1. 권장안에 미국 전용 스케줄이 빠져 있다

Claude 안은 시장별 판정을 설명하지만 실제 스케줄은 기존과 같은 `월~금 15:35 KST` 하나만 명시한다. 이 시각에는 미국의 전 거래일 종가만 확정할 수 있다. 특히 미국 금요일 종가는 토요일 오전에 확정되므로 월요일 15:35까지 기록이 늦어진다.

권장 수정:

- KRX 확정 작업과 US 확정 작업을 별도 job으로 둔다.
- KRX는 정규장 종료와 공급자 안정화 뒤 실행한다.
- US는 DST와 표준시 모두 안전한 `화~토 06:30 KST`에 실행한다.
- 두 작업 모두 실제 저장 날짜는 제공자 `price_date`를 사용한다.

### [High] C2. 최근 30일 복구는 “기존 오류 모두 수정”을 충족하지 못한다

현재 잘못된 `date.today()` 저장은 스케줄러 도입 이후 어느 날에도 발생할 수 있다. 최근 N일만 정리하면 과거 휴장일·시차 오류 행이 남는다.

권장 수정:

- 최초 운영 복구는 각 holding의 `first_buy_date`부터 최신 완료 세션까지 전체 기간을 대상으로 한다.
- 이후 자동 시작 보정만 최근 14~30일로 제한한다.
- 전체 복구는 호출량이 크므로 종목별 순차 처리, 재시도, 진행 체크포인트를 지원한다.

### [High] C3. `last_confirmed_session(market, now)`만으로 휴장일을 알 수 없다

외부 시장 달력을 쓰지 않는다면 시간과 요일만으로 마지막 거래 세션을 확정할 수 없다. 휴장일·임시 휴장·KRX 특별 거래시간을 완전히 알 수 없기 때문이다. Claude 안도 최종적으로 제공자 날짜를 신뢰한다고 쓰지만 함수 경계에서는 달력 계산처럼 보인다.

권장 수정:

- `last_confirmed_session`이 날짜를 추측하지 않게 한다.
- `safe_query_end(market, now)`는 조회 상한만 계산한다.
- 실제 세션 날짜는 `get_price_history` 또는 `PriceResult.price_date`가 반환한 바 날짜로 결정한다.
- KRX 특별 지연 폐장일을 완전히 보장해야 한다면 거래소 캘린더를 도입하거나 충분히 늦은 재확정 job을 추가한다.

### [High] C4. `max_instances=1`은 프로세스 간 중복 실행을 막지 못한다

APScheduler 3.11.2는 현재 프로세스 내부 스케줄러다. `max_instances=1`은 동일 프로세스의 겹침만 방지한다. 현재 launchd는 uvicorn 단일 프로세스라 당장 문제는 작지만, 향후 worker가 늘거나 운영자가 두 백엔드를 띄우면 동시에 실행될 수 있다.

권장 수정:

- PostgreSQL `INSERT ... ON CONFLICT (holding_id, snapshot_date) DO UPDATE`를 사용해 DB 쓰기를 원자적·멱등적으로 만든다.
- 현재 단일 프로세스 전제를 운영 문서에 명시한다.
- 다중 worker 배포 시 scheduler를 별도 프로세스로 분리하거나 PostgreSQL advisory lock을 사용하도록 후속 조건을 남긴다.

### [Medium] C5. DB와 Redis 쓰기 순서·실패 정책이 불명확하다

두 저장소는 분산 트랜잭션이 아니다. 같은 함수에서 “함께 기록”한다고만 하면 DB rollback 후 Redis만 새 값이거나, DB는 새 값인데 Redis가 과거 값인 상태가 생길 수 있다.

권장 수정:

1. 모든 성공 quote를 검증한다.
2. PostgreSQL bulk upsert를 실행하고 commit한다.
3. commit된 quote만 Redis에 set한다.
4. Redis 실패는 DB를 되돌리지 않고 경고한다.
5. Redis TTL 만료 또는 다음 조회로 회복한다.

### [Medium] C6. 종목별 부분 실패와 DB 트랜잭션 실패를 구분해야 한다

외부 API 실패는 해당 종목만 제외하면 되지만, 한 DB statement가 실패하면 세션 전체가 실패 상태가 된다. 단순한 `try/except continue`는 충분하지 않다.

권장 수정:

- 네트워크 조회 단계에서 종목별 성공/실패를 먼저 분리한다.
- 성공 결과는 검증 후 한 번의 bulk upsert로 commit한다.
- DB 오류는 전체 DB batch 실패로 명확히 보고하고 Redis를 갱신하지 않는다.

### [Medium] C7. KRX 고정 15:30 게이트는 특별 거래시간을 완전히 처리하지 못한다

미국 DST는 `America/New_York`로 해결되지만, KRX의 특별 지연 개장·폐장일은 고정 15:30만으로 판정할 수 없다. 장중 행을 절대 저장하지 않는다는 완료 조건과 충돌할 수 있다.

권장 수정 후보:

- 거래시간 달력을 도입해 세션별 close를 조회한다.
- 또는 KRX 15:45 1차 작업 뒤 충분히 늦은 시간에 재확정 job을 실행하고, UI에서 그 전까지 확정 여부를 명시한다.

이 결정은 구현 전에 명시적으로 확정해야 한다.

### [Medium] C8. 같은 시장 안의 기준일 불일치 표시 설계가 부족하다

Claude 안은 종목별 비교일을 계산한다고 했지만, 현재 API의 시장별 날짜 필드는 한 시장당 날짜 하나다. 일부 종목이 지연되면 여전히 단일 날짜가 모든 종목의 기준인 것처럼 보일 수 있다.

권장 수정:

- 시장별 `current_from/current_to`, `comparison_from/comparison_to` 범위를 반환한다.
- 범위가 같으면 단일 날짜, 다르면 날짜 범위와 `일부 종목 지연` 경고를 표시한다.

### [Low] C9. 환율 정합성은 현재 코드에서 이미 같은 응답 객체를 사용한다

Claude 안은 상단과 차트의 환율 적용이 미확정이라고 남겼다. 현재 `build_dashboard_response`는 동일한 `exchange_rate` 객체를 summary와 history에 전달한다. 이번 불일치의 직접 원인은 환율이 아니라 가격 스냅샷이다.

권장 수정:

- 이번 범위에서는 FX 저장 구조를 변경하지 않는다.
- 동일 응답 내 summary와 마지막 history가 같은 환율을 사용한다는 회귀 테스트만 추가한다.

### [Low] C10. `updated_at`만으로 과거 장중 오염을 판별할 수 없다

컬럼 추가 시 기존 모든 행에 마이그레이션 시각이 들어가므로 과거 확정 여부를 나타내지 않는다. 복구 판단은 제공자 OHLC와의 비교가 기준이어야 한다.

권장 수정:

- `updated_at`은 감사·운영 관측 용도로만 사용한다.
- 데이터 정정 여부는 복구 CLI가 제공자 거래일·종가와 비교해 결정한다.

## Codex 판정

Claude의 접근 B를 기본으로 채택하되 다음을 합쳐야 한다.

1. KRX/US 스케줄 완전 분리
2. 최초 전체 기간 데이터 복구
3. 세션 날짜 추측 금지, 제공자 거래일을 최종 기준으로 사용
4. PostgreSQL 원자적 upsert와 DB commit 후 Redis 갱신
5. 시장 내 날짜 범위 표시
6. 특별 거래시간 처리 정책 확정

이 여섯 항목을 반영해야 사용자가 요구한 “오류 모두 수정”에 도달한다.
