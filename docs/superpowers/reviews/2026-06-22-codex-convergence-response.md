## 1. C1~C10 판정 응답

### C1~C10

- C1: **동의** — 단일 월~금 KST 작업은 US 금요일 종가를 토요일에 확정하지 못하므로 US 화~토 06:30 KST 분리가 필요하다.
- C2: **동의** — `date.today()` 저장 버그는 과거 임의 날짜에 남을 수 있어 최초 1회 전체 보유 기간 복구가 완료기준 8에 맞다.
- C3: **동의** — `last_confirmed_session`이 저장일을 추정하면 안 되고, 시계 기반 함수는 조회 상한만 정하며 저장일은 provider `price_date`로 확정해야 한다.
- C4: **동의** — APScheduler `max_instances=1`은 프로세스 내부 보장일 뿐이므로 DB `ON CONFLICT` 멱등 upsert와 단일 worker 전제가 필요하다.
- C5: **동의** — DB를 진실원으로 두고 commit된 동일 quote만 Redis에 쓰며 Redis 실패는 롤백하지 않는 정책이 맞다.
- C6: **동의** — 종목별 fetch 실패와 DB batch 실패는 성격이 다르므로 성공분 검증 후 bulk upsert하고 실패를 구조화 로그로 분리해야 한다.
- C7: **동의** — KRX 고정 15:45 게이트는 수능일 같은 지연폐장에 취약하므로 별도 정책이 필요하다.
- C8: **동의** — 단일 최대 날짜로 모든 종목 기준일을 대표하면 오인 표시가 생기며, 다를 때만 경고하는 경량화가 적절하다.
- C9: **동의** — summary와 history가 같은 `exchange_rate` 객체를 쓰므로 원인은 환율 구조가 아니라 가격 스냅샷이며 회귀 테스트만 추가하면 된다.
- C10: **동의** — `updated_at`은 감사·관측용이고 과거 오염 판정의 진실원은 provider OHLC 비교와 복구 CLI여야 한다.

### 핵심쟁점 5답

- 1. US 전용 화~토 06:30 KST job: **동의** — 금요일 종가의 주말 공백을 없애고 정규·조기폐장 모두 마감 이후에 실행된다.
- 2. 최초 전체 보유 기간 복구: **동의** — 부분 기간 복구만으로는 기존 오염 행을 검증 가능하게 정리했다는 완료기준을 만족하기 어렵다.
- 3. KRX 특별 거래시간을 외부 달력 없이 안전 처리: **반대** — 멱등 교정만으로는 지연폐장 당일 장중 확정 금지를 엄격히 만족하지 못한다.
- 4. PG bulk upsert 후 Redis 갱신: **동의** — DB commit 후 Redis best-effort 갱신이 정합성과 장애 복구의 기준을 가장 명확하게 만든다.
- 5. 시장 내 기준일 범위 API 상시 노출: **동의** — 상시 4필드 노출은 과하며, 정상 케이스는 단일 날짜, 불일치 때만 경고와 종목별 날짜를 쓰면 충분하다.

## 2. K1~K3 동의 여부

- K1: **동의** — `finalize_market_snapshots`의 `total_value`는 `holding.quantity`가 아니라 거래 원장 기반 해당일 수량으로 산출하거나 컬럼 의미를 폐기해야 한다.
- K2: **동의** — 장중에는 상단 라이브 현재가와 차트 직전 확정일이 의도적으로 다를 수 있으므로 라벨과 당일 미확정 점 정책이 필요하다.
- K3: **동의** — KRX 지연폐장은 완료기준 4와 직접 충돌하므로 "나중에 덮어쓰기"만으로 최종 요구사항을 만족한다고 볼 수 없다.

## 3. KRX 특별 지연폐장 처리 — 최종 권고 확정

최종 권고는 **(b) 연1회 config override**다. 지연폐장일을 `market_close_overrides` 같은 설정으로 지정해 해당일 KRX finalize 게이트를 실제 폐장 후 버퍼 시각으로 늦추는 방식이 완료기준 4("장중 종가 확정 금지")와 정면 충돌하지 않는다.

멱등 재확정은 방어층으로 유지하되 정합성 보장의 주 수단으로 보지 않는다. override가 있는 날에는 `safe_query_end`와 쓰기 게이트가 override 시각 전까지 당일 저장을 금지하고, override 누락은 운영 체크리스트와 경고 로그로 드러내야 한다.

## 5. 남은 불확실성

pykrx가 장중 당일 OHLC를 어떤 형태로 반환하는지는 실측 테스트로 고정해야 한다.
전체 기간 복구의 호출량·레이트리밋·소요시간은 운영 데이터 규모로 산정해야 한다.
프론트의 마지막 history 점 표시가 상단 평가금액과 동일 환율·동일 수량 경로를 쓰는지 회귀 테스트로 확인해야 한다.
