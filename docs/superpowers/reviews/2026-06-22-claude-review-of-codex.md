# Claude 교차검토: Codex 시장별 종가·스냅샷 개선안

검토 대상: `2026-06-22-market-snapshot-codex-proposal.md`, `2026-06-22-codex-review-of-claude.md`(C1~C10), `2026-06-22-market-snapshot-comparison-draft.md`
근거: 공통 증거 문서 + 저장소 읽기 전용 재확인(APScheduler 버전, `build_dashboard_response` 환율 객체, 스케줄러/스냅샷 코드).

## 1. 전체 판정

Codex 독립안은 **방향이 옳고, 내 안보다 운영 구체성(시장별 스케줄·복구 범위·DB/Redis 순서·동시성)이 우수**하다. 두 안은 독립 작성에도 핵심이 거의 동일하게 수렴했으며, 차이는 충돌이 아니라 정교함이다. Codex의 C1~C10 지적 중 7건은 코드 근거로 **타당(동의)**, 2건은 방향 동의·세부 보완(**일부 동의**, C3·C8), 0건 반대다. 내 안이 추가로 기여할 부분은 (a) `total_value` 비-lot 수량 정합(R8), (b) 장중 상단≠차트의 UI 표기·차트 당일 점 처리, (c) KRX 특별폐장 잔여 엣지의 멱등 교정 정책이다. **Codex 접근 A(시장별 분리) + 내 확정 게이트(쓰기 불변식)** 결합을 권장한다.

## 4. 잠정 통합 방향 1~8 검증

1. 시장별 quote 수집·검증과 저장 분리 — **검증 통과(채택)**.
2. 제공자 거래일만 저장, 장중 당일 미확정 — **채택**. 단 "장중 금지"의 KRX 특별폐장 엣지는 §5-(3) 정책 첨부.
3. KRX/US 별도 job — **채택**(C1 근거).
4. grace + 시작 catch-up 양쪽 복구 — **채택**. 둘이 동일 확정 함수를 공유해 멱등.
5. PG 원자 bulk upsert 후 Redis — **채택**(C4·C5).
6. 전체 일회성 복구 + 짧은 자동 보정 분리 — **채택**(C2). 전체 복구에 순차·재시도·체크포인트 필수(K5).
7. 종목별 비교일 + 시장 내 날짜 범위 표시 — **부분 채택**. 범위는 "다를 때만 경고"로 경량화(C8). **추가 필요**: R8 total_value 정합(K1), 장중 상단/차트 라벨·당일 점(K2).
8. `updated_at` 관측, 진실은 OHLC 비교 — **채택**(C10).

누락 보완(통합안에 추가): **(9) `finalize`의 total_value를 원장 수량으로 산출(또는 컬럼 deprecate)**, **(10) 장중 표기/차트 당일 미확정 점 정책**, **(11) KRX 특별폐장 잔여 정책 명시**.

## 5. 핵심 쟁점 5개 답변

1. **US 전용 화~토 06:30 KST job이 필요한가?** → **필요(YES)**. US 금요일 종가 적시 확정(주말 공백 제거) + 조기폐장(13:00 ET=KST 02~03시)·정규폐장 모두 06:30 KST 이후라 안전. 저장일은 제공자 price_date.
2. **최초 전체 보유 기간 복구가 필요한가?** → **필요(YES)**. 버그가 임의 과거일에 오염을 남길 수 있어 부분 정리로는 완료기준 8 미달. 단 비용 통제를 위해 종목별 순차·재시도·진행 체크포인트로 수행. 자동 시작 보정은 14일로 분리.
3. **KRX 특별 거래시간을 외부 달력 없이 안전 처리 가능한가?** → **대부분 가능, 100% 보장은 불가**. 휴장·조기폐장은 제공자 price_date가 자동 처리(안전). 유일 취약점은 연 1회 수준 **지연폐장(수능 16:30)** 으로, 15:45 게이트가 그날 15:45~16:30을 장중인데 확정할 위험. **권장(달력 미도입)**: 멱등 upsert를 활용해 (a) 충분히 늦은 재확정 패스 또는 (b) 다음날 시작 catch-up이 종가로 덮어쓰기 + (c) UI에 "확정 전" 표기. 완전 보장이 필수 요건이면 그날만 설정 override(연 1줄). → "달력 의존성 없이 거의 안전 + 잔여 엣지는 멱등 교정"이 과설계 회피 결론.
4. **PG bulk upsert 후 Redis 갱신 순서에 반대 근거가 있는가?** → **반대 없음(타당)**. DB 진실원 + DB commit 후 Redis best-effort가 옳음. 보완 1점: 완료기준 1("마감 후 상단=차트")은 Redis set이 성공 경로에서 **동일 확정 종가**를 담아야 충족되므로, Redis 실패 시의 일시 불일치는 TTL/다음 조회로 회복됨을 문서화.
5. **시장 내 기준일 범위를 API에 노출하는 것이 과설계인가?** → **상시 4필드 노출은 부분 과설계**. 정상 운영은 같은 시장 단일 거래일이라 단일 날짜로 충분. **권장**: 같으면 단일 날짜, 다르면 `일부 종목 지연` 경고 + (보유종목에 이미 있는) 종목별 날짜로 최소 충족. `current_from/to`·`comparison_from/to` 상시 필드는 보류, 필요 입증 시 추가.

## 7. 최종 통합안에 필요한 요구사항 (우선순위)

P0 (정확성·완료기준 직결)
1. 저장 날짜 = 제공자 `price_date` (절대 `date.today()` 아님). [완료기준 5]
2. 장중 당일 바 확정 금지: `safe_query_end(market, now)`로 백필 상한 클램프 + 쓰기 경로 확정 게이트. [2,4]
3. KRX/US **분리 finalize** + 제공자 날짜 저장. US 금요일 적시 확정. [1,4]
4. PG `ON CONFLICT (holding_id, snapshot_date) DO UPDATE` 멱등·원자 upsert. [6]
5. 종가 확정 = DB commit 후 동일 값 Redis set → 마감 후 상단=차트. [1]
6. 대시보드 비교일 = 종목별 `quote.price_date` (전역 min 제거). [10]

P1 (복구·견고성)
7. 스케줄 하드닝(grace 3600, coalesce, max_instances=1) + 마감 후 시작 catch-up. [3]
8. 종목 fetch 성공/실패 분리 + 구조화 로그, DB batch 실패와 구분. [7]
9. 최초 전체기간 복구 CLI(dry-run 기본, 순차·재시도·체크포인트) + 자동 보정 14일, 진실 판정은 제공자 OHLC 비교. [8]
10. `daily_snapshots.updated_at`(server_default) — 감사·관측 전용. [9]
11. `finalize` total_value를 원장 수량 기반으로(또는 컬럼 deprecate 결정). [1 보강]

P2 (표시·관측)
12. 시장 내 날짜 동일이면 단일 표시, 다르면 `일부 종목 지연` 경고. [10]
13. 장중: 상단 "장중 현재가" 라벨 + 차트는 직전 확정일까지(당일 미확정 점 미표시). [10]
14. summary와 마지막 history가 동일 환율을 쓴다는 회귀 테스트. [1]

결정 필요(구현 전): KRX 특별 지연폐장 정책(멱등 재확정만 vs 연 1회 config override).

## 8. 남은 불확실성

1. **KRX 지연폐장(수능 등) 최종 정책** — 멱등 재확정으로 충분히 보는지, 완료기준 4의 "장중 확정 금지"를 그날까지 엄격 적용할지 팀 결정 필요.
2. **pykrx 장중 반환값 실측** — `get_market_ohlcv_by_date`의 당일 행이 현재가/직전종가/공백 중 무엇인지(게이트는 안전하나 복구 분류 정확도에 영향).
3. **전체기간 복구 비용** — 대형 계정의 제공자 호출 폭·레이트리밋·소요시간 한계 측정(체크포인트 단위 확정).
4. **차트 "마지막 평가금액" 표시 통화/시점** — 동일 `exchange_rate` 객체 사용은 확인했으나, 프론트가 마지막 history 점을 표시할 때의 통화 환산 경로가 상단과 1:1 대응하는지 표시 코드 레벨 회귀 테스트로 못 박기.
5. **US 조기폐장일 안전 논증** — 06:30 KST가 모든 조기폐장에도 안전함을 테스트(파라미터화)로 고정.
